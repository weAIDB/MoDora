# MoDora Backend 重写：架构文档

## 1. 总览

重写后的后端分为两个可独立运行的子系统，但共享同一套核心库：
- **backend-core（共享库）**：领域模型 + 核心流程编排 + 端口（接口）定义
- **backend-service（服务端）**：FastAPI 提供在线 QA 接口
- **backend-lab（实验端）**：CLI 批处理、评估与绘图

依赖方向：service/lab → core → (ports) ← infra 实现

## 2. 分层设计

### 2.1 API/CLI 层（入口层）
职责：
- 参数解析/校验
- 调用 application 服务
- 格式化输出（HTTP response / console / files）

禁止：
- 不直接调用底层模型/embedding
- 不做检索/回答业务逻辑

### 2.2 Application 层（流程编排）
建议的核心服务：
- QAPipelineService
  - parse_question（可选）
  - retrieve_evidence
  - generate_answer
  - check_answer
  - fallback_answer（whole_reason）
  - emit events（结构化日志）
- TreeBuildService
  - preprocess/build_tree 的封装
  - cache 读取/写入
- ModelRuntimeService
  - 按需加载/复用/释放模型（LLM/VLM/Embedding/OCR）
  - 预热与健康探针（可选）
- BatchRunService
  - 数据集读取、过滤、并发、断点续跑、结果写入
- EvalService
  - AIC-Acc / ACNLS 统计
  - 结果汇总与导出
- PlotService
  - seaborn/matplotlib 绘图

### 2.3 Domain 层（模型与接口）
领域模型（示例）：
- QuestionItem, Prediction, RetrievalDoc, BBox, RunMetadata, EvalMetrics
- TreeNode / CCTree（支持 SINGLE 与 MROOT）

端口接口（Ports）：
- LLMClient（text）
- VisionLLMClient（pdf/image 可选）
- EmbeddingClient
- TreeStore（tree.json load/save）
- ResultStore（jsonl 增量写入、按 questionId 覆盖策略）
- Clock/IdGenerator（用于可测性）

### 2.4 Infrastructure 层（实现）
实现模块（示例）：
- llm_remote_openai_compat：对接 OpenAI compatible API（base_url/api_key/model）
- llm_qwen_local：本地 qwen-vl 文本/多模态
- embed_qwen_local：本地 embedding
- model_registry：模型按需加载与缓存（lazy load / warmup / unload）
- ocr_engine：OCR/预处理引擎封装（可多 GPU 并行）
- store_fs_cache：cache_v2 风格的文件存储
- store_results_jsonl：结果原子写入与索引
- logging：结构化日志（json lines），run_id/request_id 贯穿

## 3. 核心数据流

### 3.1 在线 QA
1) 输入：file_name(s) + query + settings
2) TreeStore 加载单文档树或构造 MROOT
3) QAPipelineService 执行：
   - parsing（产生 location cues 与 content cues，可配置是否参与检索）
   - retrieve（返回 evidence + bboxes + node_impacts）
   - answer（模型输出）
   - check（不通过则 whole_reason 兜底）
4) 输出：Prediction + 可选事件日志

### 3.2 离线 pipeline
1) 读取 dataset（test.json 或 jsonl）
2) Filter（questionIds、pdf 范围、resume）
3) 并发执行 QAPipelineService
4) ResultStore 增量写 results.jsonl（原子写）
5) EvalService / PlotService 在 Results/ 上消费结果

## 4. 配置体系

Settings 由三部分组成：
- Paths：base_dir/cache_dir/log_dir/output_dir
- Models：qa_model、qa_answer_model、qa_parsing_model、tree_model、embedding_model
- Runtime：concurrency、remote_max_prompt_chars、use_parsed_content_for_retrieval 等

加载优先级：
1) 环境变量
2) 私有配置文件（不入库）
3) 默认值

强制要求：
- API_KEY 只能来自环境变量或私有配置；输出/日志不得包含密钥。

## 5. 兼容与迁移策略

### 5.1 结果格式兼容
- 读取：兼容 prediction 为 str 或 dict 的旧 jsonl
- 写入：统一写新 schema；必要时提供导出脚本生成旧格式

### 5.2 与现有前端兼容
- API response 字段保持 answer/retrieved_documents/node_impacts
- 多文档模式与单文档模式统一返回 retrieved_documents 的 file_name

## 6. 日志系统（分级 + 结构化 + 可追溯）

### 6.1 设计目标
- 分级别日志：DEBUG/INFO/WARNING/ERROR/CRITICAL 可配置输出阈值
- 结构化日志：关键步骤记录为可机器解析的 JSON Lines，便于聚合与定位
- 端到端追溯：service 使用 request_id，lab 使用 run_id，贯穿全链路
- 默认安全：自动脱敏，严禁输出密钥、token、Authorization 等敏感信息

### 6.2 日志形态与落地
- Console：默认输出到 stdout（便于容器化/平台采集）
- File：可选输出到文件（按天/按大小滚动，便于离线排查）
- 两类日志并存：
  - human log：面向人阅读的简洁文本（包含摘要、耗时、异常栈）
  - event log：面向机器分析的 JSON Lines（每条包含统一字段）

### 6.3 统一字段（建议）
- time、level、logger、message
- run_id（lab）、request_id（service）、trace_id（可选）
- stage（parsing/retrieve/answer/check/fallback/eval）
- model（qa_model/qa_answer_model/qa_parsing_model/embedding_model）
- dataset、questionId、pdf_id/file_names
- latency_ms、tokens_in/tokens_out（若可获取）、retry_count
- error.type、error.message（异常时）

### 6.4 组件划分
- infra/logging
  - LoggerFactory：根据 Settings 初始化 logging（level、handler、formatter）
  - Redactor：统一脱敏（key/token/url query 等）
  - Context：注入 request_id/run_id 到日志上下文（service/lab 各自实现）
- application 只产生日志事件（结构化 dict），不关心落地细节

## 7. GPU 资源管理（动态选择 + 限制占用）

### 7.1 设计目标
- 动态选择可用 GPU：根据空闲显存/排除列表/最小空闲阈值选择设备
- 限制 GPU 占用：控制并发、显存占用比例，避免跑满整机资源
- 统一管理：embedding/VL/其他 GPU 任务共享同一套资源管理器
- 可回退：无可用 GPU 时可回退 CPU 或排队等待（由配置决定）

### 7.2 核心组件
- infra/gpu
  - GpuInventory：枚举可用 GPU（数量、空闲显存、总显存）
  - GpuSelector：根据策略选择设备（most_free、round_robin、pinned 等）
  - GpuLimiter：并发限流（按“任务类型”或“设备”维度的 semaphore）
  - MemoryGuard：限制单进程显存使用比例（如 torch 的 per-process fraction）

### 7.3 选择与限制策略（建议）
- 选择策略（可配置）：
  - prefer_device：显式指定（如 "cuda:0"），优先级最高
  - exclude_devices：排除若干 GPU（如占用重要业务的 GPU）
  - min_free_gb：仅选择空闲显存≥阈值的 GPU
  - strategy：most_free / round_robin
- 并发限制（可配置）：
  - gpu_max_concurrency：全局并发上限
  - vl_max_concurrency：VL 推理并发上限
  - em_max_concurrency：Embedding 并发上限
  - per_device_concurrency：每张卡并发上限（可选）
- 显存限制（可配置）：
  - per_process_memory_fraction：单进程使用比例上限
  - oom_fallback：发生 OOM 时降低 max_new_tokens 或回退 CPU（可选）

### 7.4 与 service/lab 的集成方式
- service：每个请求在进入 GPU 任务前申请资源（acquire），完成后释放（release）
- lab：批处理按任务类型（VL/EM）统一限流，保证多线程/多进程不会超配

## 8. 模型按需加载与服务部署

### 8.1 模型按需加载（lazy load）
- 目标：冷启动轻量、按需加载、同进程内复用，避免 service/lab 启动即占满显存/内存。
- 机制：
  - 以 model_name + task_type 作为 key 的 ModelRegistry/ModelPool
  - 首次调用触发初始化（load），后续复用同一实例（或连接池）
  - 可配置预热（warmup）与释放（unload/evict），支持最大缓存数量与空闲回收
- 与 GPU 管理集成：模型加载与推理都必须在资源申请（GpuLimiter/MemoryGuard）约束下执行。

### 8.2 服务部署（service）
- 进程模型：推荐单机多 worker（与 GPU 绑定策略兼容），或单 worker + 内部并发（由配置决定）。
- 探针：
  - health：进程存活、配置加载、日志初始化可用
  - readiness（可选）：远程 LLM 连通性、关键模型可按需加载（不强制预热）
- 可观测性：每次请求生成 request_id，关键阶段产出结构化 event log，便于线上排障与离线复盘。

### 8.3 多 GPU 并行（OCR/预处理与实验）
- 预处理/OCR：将 PDF/页面划分为可独立处理的任务单元，按 GPU 维度分发，保证单卡并发可控。
- 实验端：批量推理支持多 GPU 并行（多进程或 worker pool），并与统一限流器/选卡策略一致。
