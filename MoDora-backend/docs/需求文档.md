# MoDora Backend 重写：需求文档

## 1. 背景与目标

### 1.1 背景
- 现有后端（MoDora-main）同时承载“服务端接口”和“离线实验/批处理”，并在模块内混合了配置、模型调用、检索、评估与落盘逻辑，维护与复现成本较高。

### 1.2 目标
- 将“后端服务”和“实验/评估”拆分为两个明确边界的子系统：
  - **服务端（backend-service）**：提供在线 QA API、树构建/管理、运行日志与检索证据返回。
  - **实验端（backend-lab）**：提供离线 pipeline、断点续跑、批量评估、图表生成、结果导出。
- 实现统一的数据结构（输入/输出/中间产物），保证：
  - 服务端与实验端复用同一核心能力（解析→检索→回答→校验/兜底→记录）。
  - 不同模型后端（本地/远程）可通过配置切换，不改业务代码。
- 把敏感配置（API_KEY 等）彻底从代码中移除，只允许环境变量/私有配置注入。

## 2. 范围界定

### 2.1 本期范围（必须实现）
- 数据结构与协议
  - 定义统一的 Result JSONL schema（可兼容读取旧格式）。
  - 明确 RetrievalDoc、Prediction、RunMetadata、EvalMetrics 等结构。
- 服务端（API）
  - 单文档与多文档（MROOT）两种模式的 QA 请求。
  - 返回 answer + retrieved_documents + node_impacts（与前端当前展示需求对齐）。
  - 可选返回结构化事件日志（用于 debug/复现）。
- 模型与部署
  - 支持模型按需加载（lazy load）：默认不在进程启动时加载全部模型，请求触发加载并缓存复用。
  - 支持服务部署：提供标准启动方式与健康检查，便于在服务器/容器环境运行。
- 实验端（CLI）
  - 对数据集（如 MMDA test.json）运行 pipeline，支持：
    - 只跑指定 questionIds
    - 断点续跑（resume）
    - 并发与限流
  - OCR/预处理与实验推理支持多 GPU 并行分发（与并发/限流策略兼容）。
  - 对 Results 目录批量统计 AIC-Acc(%) 与 ACNLS Score，并生成图表（matplotlib + seaborn）。
- 存储与缓存
  - tree.json/cp.json 等缓存读写封装为可替换组件。
  - 结果写入具备原子性与幂等性（按 questionId 去重/覆盖策略明确）。

### 2.2 非本期范围（暂缓）
- 前端改造（MoDora-frontend 不动，仅保证接口兼容）。
- 数据持久化
- 用户管理

## 3. 用户故事（关键用例）

### 3.1 服务端用户故事
- 作为前端用户，我选择 1 个或多个 PDF，并提问，系统返回：
  - 回答
  - 证据片段（页码/文本/bboxes/文件名）
  - 命中节点影响值（用于树高亮）
- 作为调试人员，我能在响应中查看：
  - question parsing 的输出（位置 cues、内容 cues）
  - 检索阶段命中的节点与证据数
  - answer 的 check 是否通过、是否触发兜底 whole_reason

### 3.2 实验端用户故事
- 作为研究者，我用一条命令对数据集批量跑 inference:
  - 支持断点续跑，得到 results.jsonl。
  - 有详细的实验日志方便研究
- 作为研究者，我把多个模型的 results.jsonl 放入 Results/，一条命令得到：
  - 每个文件的 AIC-Acc(%) 与 ACNLS Score 汇总
  - seaborn/matplotlib 生成的对比图


## 4. 数据与格式要求

### 4.1 结果 JSONL 最小字段（统一 schema）
每行是一个 dict，至少包含：
- questionId: int
- question: str
- pdf_id: str（或 file_names: list[str]）
- answer: list[str]（reference answers）
- tag: str（可选）
- prediction: object
  - answer: str
  - retrieved_documents: list[object]
    - file_name: str
    - page: int
    - content: str
    - bboxes: list[object]（范围/页码/source_path 可选）
  - node_impacts: object（key: str, value: int）
- judge: str（可选，若已评估）

### 4.2 指标定义（与现有 ai_evaluate.py 对齐）
- AIC-Acc(%)
  - 基于 judge 字段：包含 "T" 视为正确。
- ACNLS Score
  - 优先做 reference containment（reference in prediction），命中则计 1；
  - 否则按 NLS（1 - edit_distance/max_len），阈值 0.5，低于阈值记 0；
  - 最终对所有样本取平均。

## 5. 非功能性需求
- 可维护性
  - 清晰分层（入口/API、application、domain、infra），避免跨层依赖倒置。
  - 模型/检索策略以配置驱动。
- 可复现性
  - 每次 run 生成 run_id，并记录 settings（不含密钥）、模型名、数据集路径、代码版本标识（可选）。
- 性能与稳定性
  - 支持并发批处理与限流，避免 GPU/远程 API 过载。
  - 支持动态选择可用 GPU（按空闲显存/排除列表/最小阈值等策略）。
  - 支持限制 GPU 使用（并发上限、每进程显存比例、OOM 降级策略）。
  - 支持 OCR/预处理与实验推理的多 GPU 并行（跨设备调度、每卡限流、任务隔离）。
  - 支持模型按需加载与复用（冷启动轻量、请求触发加载、可配置预热与回收策略）。
  - 关键步骤输出结构化日志，可定位 parsing/检索/回答/兜底哪个环节失败。
- 日志与可观测性
  - 支持分级日志（DEBUG/INFO/WARNING/ERROR/CRITICAL）与统一格式。
  - 支持结构化事件日志（JSON Lines），包含 run_id/request_id、阶段耗时、模型名等字段。
