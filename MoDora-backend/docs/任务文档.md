# MoDora Backend 重写：任务文档

## 0. 交付目标
- 可部署：backend-service 启动 FastAPI，前端不改即可调用
- 可实验：backend-lab 一条命令跑数据集，一条命令评估+出图
- 可复现：每次 run 生成 run_id，记录 settings（不含密钥）

## 1. 里程碑与验收

### M1：工程骨架与配置（1–2 天）
任务：
- 建立 core/service/lab 三层工程结构
- 引入 Settings（env + config 文件）与密钥管理（代码中不出现 key）
- 建立统一日志初始化（分级、console/file、脱敏、run_id/request_id 注入）
验收：
- `service` 能启动并返回 health
- `lab` 能运行 `--help`

### M2：统一数据模型与结果读写（2–3 天）
任务：
- 定义统一 Result JSONL schema（QuestionItem/Prediction/RetrievalDoc）
- 实现 ResultStore（增量写、原子写、按 questionId 覆盖/跳过策略）
- 实现旧结果读取 adapter（prediction 兼容 str/dict）
验收：
- 能读取现有 Results/*.jsonl 并转换为统一对象
- 能输出统一格式 results.jsonl

### M3：模型与 embedding 端口 + 实现（3–5 天）
任务：
- 定义 LLMClient / EmbeddingClient 接口
- 实现 RemoteOpenAICompat、QwenLocal、QwenEmbedding
- 引入 GPU 资源管理器（动态选卡、并发限流、显存限制、OOM 降级）
验收：
- 单条 query 能在本地/远程两种模式下成功返回
- 日志记录模型名与耗时

### M4：TreeStore 与缓存体系（3–5 天）
任务：
- TreeStore：加载/保存 tree.json，支持 SINGLE 与 MROOT
- TreeBuildService：封装 preprocess/build_tree
验收：
- 单文档树可构建、可加载
- 多文档 MROOT 可构建并参与检索

### M5：检索子系统（5–10 天）
任务：
- 拆分 LocationFilter / TitleSelector / Verifier / (EmbeddingFallback 可选)
- 以策略方式组合 Retriever（配置开关）
- 产出 retrieved_documents（按 file+page 去重策略可配置）
验收：
- 对固定样例输出稳定（证据数量、页码合理）
- 记录 node_impacts

### M6：QA 编排服务（3–6 天）
任务：
- QAPipelineService：parse→retrieve→answer→check→fallback
- 结构化事件日志（每阶段输入/输出摘要、耗时、fallback 是否触发）
验收：
- 与现有前端所需字段对齐：answer/retrieved_documents/node_impacts
- debug 时可追溯 parsing 输出与 check/fallback 结果

### M7：批量 pipeline + 断点续跑（3–6 天）
任务：
- CLI：run dataset，支持 questionIds 过滤与 resume
- 并发执行与失败重试策略（可配置）
验收：
- 可在中断后继续跑，不重复写入已完成 questionId

### M8：评估与绘图（1–3 天）
任务：
- EvalService：AIC-Acc/ACNLS（对齐旧实现）
- PlotService：matplotlib+seaborn 输出对比图（Results 目录）
验收：
- 对 Results 目录生成 metrics.json + metrics.png

## 2. 风险与对策
- 模型行为差异导致结果漂移：用“固定样例集 + 快照输出”做回归
- 并发导致显存/速率爆：所有资源使用统一限流器，默认保守配置
- 旧数据格式混乱：adapter 层隔离兼容逻辑，core 一律使用统一对象

## 3. 建议的开发顺序
- 先 M1+M2（地基）→ 再 M3（模型接口）→ 再 M6（跑通一条）→ 再 M5（优化检索）→ 最后 M7/M8（规模化与评估）
